🛡️ VYPER — Smart Contract Bug Hunter

VYPER is a local-first smart contract security auditing platform. It runs 28 microservices on a single laptop via Docker Compose. Scan Immunefi contracts, find True Positive bugs, generate exploit PoCs, and produce submission-ready reports — all in one automated pipeline.

🎯 Bug Bounty Hunter

Scan 234+ Immunefi programs, detect True Positives, auto-generate submission-ready reports.

🔬 Security Research

Exploit findings with Anvil fork engine. Prove vulnerabilities with real on-chain execution.

📊 Platform Metrics

Track precision per tool (Slither, Mythril, Echidna, Halmos, Manticore). Learn from False Negatives.

🤖 AI-Powered

ANTONIO — the absolute AI agent controller with ReAct loop, vector memory, and autonomous hunting.

🚀 Quick Start

Prerequisites

  • Docker & Docker Compose installed
  • 4+ CPU cores, 8+ GB RAM, 20+ GB disk

1. Clone & Setup

git clone <repo-url> sc_auditor
cd sc_auditor
cp .env.example .env   # Optional: add API keys (OpenAI, Anthropic, Etherscan)

2. Start All Services

# Build & start all 28 services
docker compose up --build -d

# Check health
docker compose ps

# View logs
docker compose logs -f 15-dashboard
docker compose logs -f 11-orchestrator

3. Access Dashboard

http://localhost:8000

4. Run an Audit

# Start audit via Orchestrator API
curl -X POST http://localhost:8009/audit/start \
  -H "Content-Type: application/json" \
  -d '{"address": "0x4c9edd...", "chain": "ethereum"}'

# Check status
curl http://localhost:8009/audit/{audit_id}/status

5. Scale Scanners

# Run 3 scanner instances in parallel
docker compose up --scale 04-scanner=3 -d

Scanner Isolation

Each scanner runs as an independent microservice. Benefits:

  • Fault isolation: Slither crash doesn't affect Mythril or Echidna
  • Independent scaling: Run 3 Slither instances for parallel scans
  • Resource limits: Mythril (heavy) gets 2 CPU/4GB RAM, Forge (light) gets 0.5 CPU/512MB
  • Independent updates: Upgrade Manticore without touching other scanners

🔗 Communication Patterns

HTTP/REST (Primary)

All inter-service communication uses HTTP/REST via Python httpx (async). No message queues, no gRPC — pure HTTP for simplicity and debuggability.

PatternUse CaseTechnology
Sync RESTService-to-service API callshttpx (async) + FastAPI
SSEReal-time pipeline progress to UIServer-Sent Events (Dashboard ↔ Orchestrator)
Docker APIAnvil container lifecycledocker-py via /var/run/docker.sock

Config Service Pattern

Every service reads configuration from 01-config via HTTP at startup:

config = httpx.get("http://01-config:8000/config").json()

Config Service is the only shared dependency. No service reads another service's data directly — all access goes through APIs.

⚙️ Audit Pipeline

Each audit goes through 8 stages, orchestrated by 11-orchestrator with retry (3 attempts), saga pattern rollback, and configurable timeouts.

┌─────────────┐ │ PENDING │ └──────┬──────┘ │ start ▼ ┌─────────────────────┐ │ FETCHING_PROGRAM │ ← 02-immunefi └──────────┬──────────┘ │ ▼ ┌─────────────────────┐ │ FETCHING_SOURCE │ ← 03-source └──────────┬──────────┘ │ ▼ ┌─────────────────────┐ │ SCANNING │ ← 04-router → 04a/b/c/d/e + 05 │ Slither + Mythril │ (all 6 tools in parallel) │ + Echidna + Halmos │ │ + Manticore │ └──────────┬──────────┘ │ ▼ ┌─────────────────────┐ │ AI_ANALYSIS │ ← 06-ai (LLM verdict) └──────────┬──────────┘ │ ▼ ┌─────────────────────┐ │ CLASSIFYING │ ← 07-classifier (TP/FP) └──────────┬──────────┘ │ ▼ ┌─────────────────────┐ │ EXPLOITING │ ← 08-exploit (Anvil) │ (only if TP │ │ critical/high) │ └──────────┬──────────┘ │ ▼ ┌─────────────────────┐ │ REPORTING │ ← 09-reporter └──────────┬──────────┘ │ ▼ ┌─────────────────────┐ │ NOTIFYING │ ← 10-notifier └──────────┬──────────┘ │ ▼ ┌─────────────────────┐ │ COMPLETED │ ✓ └─────────────────────┘ States: pending → fetching_program → fetching_source → scanning → ai_analysis → classifying → exploiting? → reporting → notifying → completed

💾 Storage Architecture

JSON File-Based Storage

No database engines. All data stored as JSON files in Docker volumes. The only exception is 17-experience which uses SQLite for structured learning data.

├── immunefi/programs/{slug}.json   # Per program (234+)
├── source/contracts/{chain}/{addr}/# Source cache
├── scanner/results/{audit_id}/     # Scan results per tool
├── ai/cache/{hash}.json            # LLM response cache
├── classifier/metrics.json         # TP/FP/TN/FN metrics
├── exploit/results/{id}/           # PoC scripts + results
├── reporter/reports/{audit_id}/    # immunefi.md + full.md

Data Ownership Rules

  • Each service has its own Docker volume
  • Services never read another service's files directly
  • Config read-only for all services (accessed via HTTP)
  • Shared volumes: vyper_learning (classifier + orchestrator), vyper_kb (classifier + exploit)

🏷️ Bug Classification System

The 4-Quadrant Detection Matrix

Every finding is classified into 4 categories using the confusion matrix framework:

ACTUAL: YES (Bug)
ACTUAL: NO (Safe)
DETECTED: YES
TRUE POSITIVE (TP)
✅ Real bug — Submit to Immunefi
FALSE POSITIVE (FP)
❌ False alarm — Record, adjust patterns
DETECTED: NO
FALSE NEGATIVE (FN)
⚠️ MISSED! — MOST IMPORTANT — Create new pattern
TRUE NEGATIVE (TN)
✅ Correctly safe — Increase confidence

Finding Lifecycle

  1. RAW (from Scanner) → classification: unknown
  2. AI VERDICT (06-ai) → TP (confirmed) / FP (rejected)
  3. EXPLOIT TEST (08-exploit) → exploit success = TP confirmed
  4. HUMAN REVIEW (Dashboard) → final classification
  5. IMMUNEFI SUBMISSION → accepted = TP final / rejected = reclassify
  6. LEARNING LOOP → TP reinforces pattern, FN triggers new pattern creation

Dual Reporting

immunefi.mdBug bounty submissionTP-only, PoC required, Immunefi format

🤖 Antonio — AI Agent Controller

Antonio (14-agent) is the single absolute AI agent controller. All services, all agents, all pipelines operate under Antonio's command. No other service has AI autonomy.

Chain of Command

LEVEL 0: USER (Human Operator)
  │ Only communicates with ANTONIO
  │ NO direct access to service agents
  │ Input: "Antonio, audit contract 0x..."
  ▼
LEVEL 1: ANTONIO (14-agent) — ABSOLUTE CONTROLLER
  │ Only AI Agent with full ReAct loop
  │ Single owner of AgentRegistry
  │ Controls all delegation (DelegateTaskSkill)
  │ Centralized memory (vector + episodic + semantic + graph)
  │ Central learning (FeedbackLearner)
  ▼
LEVEL 2: ORCHESTRATOR (11) — Antonio's subordinate
  │ Pipeline state machine — NO AI autonomy
  │ Daemon only active when Antonio/Dashboard enables it
  ▼
LEVEL 3: SERVICE AGENTS (02-23, except 14)
  │ All are BaseAgents — receive delegation, no initiative
  │ No access to Antonio's AgentRegistry
  │ No AI ReAct loop — pure task executors

💥 Exploit Engine

The Exploit Engine (08-exploit) manages ephemeral Anvil Docker containers to prove vulnerabilities through real on-chain execution.

Capabilities

  • Fork mainnet/testnet at any block number
  • Impersonate accounts (owner, whale, governance)
  • Snapshot/reset state for multiple exploit variants
  • Manipulate balance, storage, timestamp
  • Debug trace to opcode level
  • Generate PoC scripts in Hardhat/Foundry format

❓ FAQ

Does VYPER need internet?

Partially. Services that need internet: AI (OpenAI/Anthropic API), Immunefi (GitHub sync), Source (Etherscan/GitHub fetch). All other services run fully offline.

Can it scan Arbitrum/Polygon/BSC contracts?

Yes. Source Service supports 5 providers: Etherscan, Sourcify, GitHub, Blockscout, and Manual upload. EVM chain support includes: Ethereum, Arbitrum, Optimism, Base, Polygon, BNB Chain, Avalanche, Fantom, Mantle.

How do I submit to Immunefi?

The pipeline generates immunefi.md — ImmuneFi-standard format with TP-only findings, severity classification, PoC, impact analysis, and fix recommendations. Ready to copy-paste to Immunefi dashboard. The 16-submission service can also handle semi-automated submission.

Can I scale VYPER?

Yes. All services are stateless and scalable:

docker compose up --scale 04-scanner=3 -d

What's the difference between the 6 scanners?

ScannerBest For
SlitherFast static analysis — control flow, reentrancy, inheritance bugs
MythrilDeep symbolic execution — complex path exploration
EchidnaFuzzing — property-based testing, invariant checking
HalmosFormal verification — mathematical proof of properties
ManticoreSymbolic execution focused on HIGH/CRITICAL severity
ForgeBuild verification — compilation + test runner

What is Antonio?

Antonio is the sole AI agent controller (14-agent). It has a ReAct loop, vector memory, and controls all delegation. No other service has AI autonomy — Antonio is the single brain of VYPER.

Does VYPER support non-EVM chains?

Yes, StarkNet/Cairo support was added June 2026 via services 22-source-starknet and 23-scanner-cairo.
